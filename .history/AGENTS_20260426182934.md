# AGENTS.md

## 项目约定

- 本项目根据 `Study/基本设计.md` 实现，需求变更先更新设计文档或在 `docs/` 中记录。
- 不硬编码账号、密码、IP、域名、密钥和支付参数；统一通过环境变量注入。
- 后端使用 Spring Boot 3 + MyBatis Plus + PostgreSQL。
- 管理后台和学生 Web 使用 Vue3 + Vite + TypeScript + Element Plus。
- 手机端使用 UniApp，首期优先 H5，后续按小程序/App 目标适配。
- 所有项目配置必须区分 `dev`、`stg`、`prod` 三套环境。

## 后端规范

- Spring Boot 配置使用 profile 分层：`application-dev.yml`、`application-stg.yml`、`application-prod.yml`。
- 后端 PostgreSQL 连接池统一使用 HikariCP，并在 `dev/stg/prod` 中分别配置连接池参数。
- 业务代码不允许手动创建数据库连接，统一通过 Spring DataSource/MyBatis Plus 管理。
- 后端按业务模块组织代码，每个模块优先包含 `controller`、`dto`、`vo`、`entity`、`mapper`、`service`、`service/impl`。
- Controller 只负责接收请求、参数校验、调用 Service 和返回结果，不写业务编排和数据库逻辑。
- Service 负责业务规则、事务边界、权限上下文校验和跨表操作；Mapper 只负责数据库访问。
- 每张业务表建议对应 `Entity`、`Mapper`、`Service`、`ServiceImpl`。
- 通用 CRUD 使用 MyBatis Plus；复杂统计、班级报表、排行榜使用自定义 Mapper SQL。
- 入参使用 `DTO`，出参使用 `VO`，数据库映射使用 `Entity`；`Entity` 不直接返回给前端。
- `DTO` 必须使用 Bean Validation 声明必填、长度、格式、范围、枚举等约束，Controller 使用 `@Valid` 或 `@Validated` 触发校验。
- 分页查询统一使用 `page`、`pageSize`，分页响应统一包含 `records`、`total`、`page`、`pageSize`。
- Controller 返回统一响应结构 `ApiResponse<T>`。
- API 失败响应必须使用稳定错误码，不能让前端依赖中文文案判断业务逻辑。
- 所有受限接口必须在后端校验权限，不能只依赖前端隐藏菜单。
- 后台接口按认证、角色、权限点三层控制，权限点命名建议使用 `模块:资源:动作`。
- 后端必须使用 `@RestControllerAdvice` 做全局异常处理，Controller/Service 不要为了返回错误响应而写大量局部 `try/catch`。
- 业务错误统一抛出 `BusinessException`；只有补偿、资源释放、重试、追加上下文或转换第三方异常时才局部捕获。
- 未预期异常必须记录日志并返回通用错误信息，不能向前端暴露堆栈、SQL、密钥或内部路径。
- 多表写入、支付状态变更、会员权益调整、班级成员审核等必须在 Service 层使用 `@Transactional` 管理事务。
- 事务内避免调用慢接口、文件上传、第三方支付请求等外部不确定耗时操作。
- 后台管理端和学生端统一使用 JWT 认证；登录成功后后端签发 Token，前端通过 `Authorization: Bearer <token>` 携带。
- JWT 密钥、过期时间、签发方等配置必须通过环境变量注入，禁止写死在代码或配置仓库中。
- 后台接口必须校验 JWT、管理员状态、角色和接口权限；管理员禁用、强制下线等场景后续可结合 Redis/Token 黑名单处理。
- 日志必须包含请求追踪 ID；禁止打印密码、JWT、支付密钥、银行卡号、验证码等敏感信息。
- 支付回调、会员手动调整、后台关键操作必须记录审计日志。
- 缓存只用于性能优化，核心状态以 PostgreSQL 为准；Redis key 必须统一前缀、设置 TTL，并在数据变更后清理或更新。
- 上传文件必须校验大小、后缀、MIME 类型和业务归属，存储 key 由后端生成，文件元数据必须落库。
- 慢操作、通知、统计汇总、文件处理和第三方重试优先异步化；异步任务必须保证幂等。
- 核心 Service、复杂 Mapper、支付回调、会员权益、班级成员审核和权限校验必须优先补测试。

## 前端规范

- 前端配置使用 Vite mode 分层：`.env.dev`、`.env.stg`、`.env.prod`，仓库只提交对应 `.example` 文件。
- Vue 单文件组件统一使用 Composition API 和 `<script setup lang="ts">`，组件 props、emits 必须声明 TypeScript 类型。
- 页面按业务模块放入 `src/views`，公共组件放入 `src/components`，组合式逻辑放入 `src/composables`。
- API 调用统一放入 `src/api`，禁止在页面组件中直接散落 `fetch`、`axios` 或请求地址。
- 类型定义放入 `src/types`，通用工具放入 `src/utils`，样式入口放入 `src/styles`。
- 所有请求必须经过统一请求封装，统一处理 baseURL、超时、JWT、错误提示、401/403、后端统一响应结构。
- API baseURL 只能使用 `VITE_API_BASE_URL` 等 Vite 环境变量，不能硬编码 IP、域名或环境地址。
- 前端按后端 `ApiResponse` 处理 `success`、`code`、`message`、`data`、`traceId`、`timestamp`；失败判断以稳定错误码为准。
- 路由统一在 `src/router` 管理，路由组件使用懒加载，后台管理端路由建议配置 `requiresAuth`、`roles`、`permissions`、`title` 等 meta。
- 状态管理使用 Pinia，按业务域拆分 Store；Store 只保存跨页面共享状态，组件临时状态保留在组件内。
- 需要持久化的状态必须明确范围和过期策略，禁止把敏感信息、完整用户隐私数据长期放入 localStorage。
- 管理后台登录后保存 JWT，所有后台 API 请求必须自动附加 `Authorization: Bearer <token>`；退出登录必须清理 Token。
- 退出登录必须同时清理用户信息、菜单权限和相关缓存状态；401 统一跳转登录或刷新登录态，403 显示无权限提示。
- 表格、表单、弹窗优先使用 Element Plus 原生组件，避免引入大型 UI 依赖。
- 表单必须使用 Element Plus Form rules 或等价机制做前端校验，同时不能替代后端校验。
- 新增、编辑、审核、删除、支付相关操作必须有明确确认、成功、失败反馈，并处理提交中和防重复提交。
- 列表页必须具备加载中、空数据、错误、分页、筛选重置等状态。
- 上传组件必须在前端校验文件大小、后缀、MIME 类型，并由后端再次校验。
- 前端枚举和后端状态值必须和 `database/ENUMS.md` 对齐，不能自行创造状态字符串。
- 默认不使用 `v-html`；确需富文本展示时必须经过后端或可信库清洗。
- 日志、埋点、错误上报不能记录 JWT、密码、验证码、支付参数等敏感信息。
- 管理后台如果采用 vue-pure-admin 精简版（国际化版），必须沿用模板的 i18n、路由、菜单和权限约定。
- 需要多语言的页面，显示文案统一进入 i18n 资源文件，避免散落硬编码。
- 提交前必须执行 `pnpm typecheck` 和对应环境的 `pnpm build`；后续接入 ESLint、Prettier、Vitest、Playwright 时需保证可在 CI 中执行。

## UniApp 规范

- 手机端使用 UniApp + Vue3 + TypeScript，首期优先 H5，后续按小程序/App 目标逐步适配。
- 移动端请求、认证、错误处理要和 Web 前台保持一致，不重复造一套业务协议。
- 页面统一放在 `src/pages`，公共组件放入 `src/components`，组合式逻辑放入 `src/composables`。
- API 请求统一放入 `src/api`，状态管理放入 `src/stores`，类型定义放入 `src/types`，工具方法放入 `src/utils`。
- 静态资源放入 `src/static` 或项目约定静态目录，大体积图片、音频、视频优先使用对象存储 URL，不放入主包。
- 页面路由、原生导航栏、tabBar、分包、预下载统一在 `pages.json` 管理，不在代码中硬编码页面路径。
- `manifest.json` 只配置当前目标平台真正需要的能力、权限、SDK 和模块，禁止提前开启无用权限。
- 小程序平台涉及网络请求、上传、下载、业务域名等必须提前在对应平台配置白名单。
- 跨端差异优先通过 UniApp 标准 API 抹平；只有确实存在平台能力差异时才使用条件编译。
- 条件编译代码必须尽量集中在适配层、工具层或平台专属组件中，业务页面只调用统一接口。
- 网络请求统一封装 `uni.request`，统一处理 baseURL、timeout、JWT、错误码、401/403、loading 和后端 `traceId`。
- 小程序和 App 的接口域名必须使用 HTTPS；本地调试例外要限制在 `dev` 环境。
- 请求超时时间、API 地址、上传地址必须走 Vite/UniApp 环境变量或平台配置，不能硬编码。
- 登录成功后统一保存 Token；请求头使用 `Authorization: Bearer <token>`。
- 401 统一清理登录态并跳转登录页；403 显示无权限提示；网络异常提供重试入口。
- `uni.setStorage` / `uni.getStorage` 必须通过统一缓存工具封装，缓存 key 使用项目统一前缀。
- Token、用户信息、学习进度、临时草稿等缓存必须明确 TTL 或清理时机。
- 学习记录、做题记录、背词进度等离线数据必须设计同步状态，避免重复提交或覆盖服务端数据。
- 退出登录、切换账号、Token 失效时必须清理账号相关缓存。
- 初始化页面参数使用 `onLoad`，每次进入页面需要刷新状态的逻辑使用 `onShow`；`onShow` 会多次触发，不能无节制重复请求。
- 定时器、音频播放、事件监听、订阅消息等必须在 `onHide` 或 `onUnload` 中释放。
- 页面跳转统一封装导航工具，避免散落 `uni.navigateTo`、`uni.redirectTo`、`uni.switchTab` 的硬编码路径。
- 移动端布局优先使用 `rpx`、flex 和安全区适配，避免固定像素导致小屏溢出。
- 列表页必须有加载中、空数据、错误、下拉刷新、触底加载和重试状态。
- 图片、音频、视频上传前必须校验大小、后缀、MIME 类型、数量和业务归属。
- 上传统一走封装方法，统一处理 Token、进度、取消、失败重试和后端返回结构。
- 音频播放、录音、相册、相机等能力必须在使用前检查平台权限并给出用户可理解的提示。
- 小程序和 App 必须控制主包体积，学习、班级、会员、媒体等非首屏模块优先分包。
- 首页只加载首屏必要数据和资源，长列表必须分页或虚拟化，图片必须懒加载并压缩到合适尺寸。
- App 端必须配置隐私政策提示，并在用户同意前避免调用定位、录音、相机、相册等敏感权限 API。
- iOS App 发布前必须检查隐私清单、权限用途说明和第三方 SDK 隐私声明。
- Android App 发布前必须检查权限申请时机、权限用途说明和应用市场隐私合规要求。
- 不使用的 SDK、模块和权限必须从 `manifest.json` 移除。
- 提交前至少执行 `pnpm typecheck` 和 `pnpm build:h5`。
- H5、小程序、App 的关键流程必须分别验证，不能只用 H5 结果代表所有平台。
- 登录、学习记录、音频播放、文件上传、会员购买、班级审核、支付跳转属于重点回归路径。

## 数据库规范

- 数据库使用 PostgreSQL。
- 迁移脚本放入 `database/migrations`。
- 数据库结构变更默认使用 Flyway 管理，迁移脚本命名格式为 `V版本号__说明.sql`。
- 已进入共享环境的迁移脚本不可修改，只能新增下一版本迁移。
- `stg`、`prod` 是否由应用启动自动迁移，必须通过环境变量控制，并经过发布流程确认。
- 表名使用小写蛇形命名。
- 所有业务表保留 `created_at`、`updated_at`，状态字段使用明确枚举文本。
- DDL 必须包含主键、索引、字段注释。
- 外键关系必须创建 FK，不能只靠字段名或文档约定。
- 状态值、枚举值必须单独整理到 `database/ENUMS.md`，并在 DDL 中尽量使用 `check` 约束保护。

## 提交前检查

- 后端：`mvn test`
- 管理后台：`pnpm typecheck && pnpm build`
- 学生 Web：`pnpm typecheck && pnpm build`
- 确认 `.env`、密钥、真实支付参数没有提交。
